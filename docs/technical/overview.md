## 2. 项目目标

明叶同行 v1 是一个轻量拼车信息平台，产品形态为：

- H5 业务端：面向车主与乘客，支持浏览、发布、收藏、联系、个人中心
- PC 管理端：面向管理员，支持登录、看板、行程管理、用户查询
- Backend：统一承载 H5 与 Admin 的业务接口、文件上传和定时任务

v1 的核心目标是先完成信息发布、浏览、联系和运营管理闭环，不引入支付、评价、地图、复杂中台能力。

## 3. 技术选型

### 3.1 总体选型

| 模块 | 技术方案 |
|------|----------|
| 后端语言 | Go |
| 后端框架 | Gin |
| 配置管理 | Viper |
| 日志 | Zap + lumberjack |
| 数据库 | MySQL 8.x |
| 对象存储 | Cloudflare R2 |
| H5 前端 | React + Vite |
| Admin 前端 | React + Vite + Ant Design |
| 鉴权方案 | JWT 双令牌（纯无状态） |

### 3.2 架构原则

- 采用单体分层架构，不做微服务拆分
- H5、Admin 共用同一个后端服务，但通过路由域和鉴权域隔离
- 后端在现有 `mn-backend` 骨架基础上扩展，不推翻现有 Gin / 配置 / 日志结构
- v1 不引入 Redis，不做 token 黑名单，不做服务端会话存储
- v1 不引入消息队列、搜索引擎、复杂缓存体系

## 4. 系统架构

### 4.1 系统组成

```text
mn-frontend-h5       -> H5 业务端
mn-frontend-admin    -> PC 管理端
mn-backend           -> 统一业务服务
MySQL                -> 业务主数据
Cloudflare R2        -> 头像等文件资源
```

### 4.2 路由域划分

- H5 业务接口：`/api/v1/...`
- Admin 接口：`/api/admin/v1/...`

两套接口共享数据库，但认证体系严格隔离：

- H5 只接受 `role=user` 的 token
- Admin 只接受 `role=admin` 的 token

## 5. 认证与鉴权

### 5.1 统一规则

- 认证方案：JWT 双令牌
- `accessToken` 有效期：2 小时
- `refreshToken` 有效期：72 小时
- Token 为纯无状态 JWT，自包含，不落库
- 服务端不存储会话，不使用 Redis，不做黑名单
- 允许多设备同时登录
- 退出登录仅清理客户端本地 token

### 5.2 Token 约定

建议 claims 至少包含：

- `sub`：主体 ID
- `role`：`user` / `admin`
- `type`：`access` / `refresh`
- `iat`
- `exp`

H5 侧当前约定：

- 登录成功后持久化 `accessToken / refreshToken`
- 业务接口返回 `1006` 或 `1007` 时，客户端自动调用 `POST /api/v1/auth/refresh`
- refresh 成功后重放原请求，失败后清理本地登录态并跳转登录

### 5.3 能力边界

由于采用纯无状态 JWT，v1 明确不支持：

- 退出后服务端立即失效
- 修改密码后旧 token 立即失效
- 单设备登录
- 管理员强制踢下线

这些限制必须同步给前后端开发，不得误判为系统缺陷。

## 6. 核心业务模型

v1 核心模型统一为以下五类：

- `users`：H5 用户
- `admins`：管理员
- `trips`：行程
- `trip_favorites`：收藏关系
- `file_assets`：文件资源

建模原则：

- 车找人与人找车统一建模为 `trip`
- 行程状态统一使用 `active / full / closed / expired`
- 联系方式存为行程发布时的快照，不跟随用户资料变更
- 收藏记录独立存储，便于保留历史关系

---

## 7. 共享业务规则

### 7.1 行程状态规则

| 状态 | 含义 | 维护方式 |
|------|------|----------|
| `active` | 正常可见可联系 | 用户或管理员创建/编辑 |
| `full` | 已满员 | 用户或管理员修改 |
| `closed` | 手动关闭/下线 | 用户或管理员修改 |
| `expired` | 已过期 | 服务启动即执行一次，之后由服务端定时任务更新 |

约束：

- `expired` 为系统态，用户和管理员都不应直接写入
- 已过期行程默认不允许恢复为有效态

### 7.2 收藏规则

- 收藏接口对外表现为 toggle
- 底层通过新增/删除 `trip_favorites` 记录实现
- 行程下线后收藏记录保留
- 收藏列表按收藏时间倒序

### 7.3 文件上传规则

- 头像上传使用对象存储
- 支持格式：`jpg / png / webp / heic`
- 最大 10 MB
- 上传成功后返回完整 URL

## 8. 日志与可观测性

后端统一记录四类日志：

- 访问日志
- 业务日志
- 错误日志
- 定时任务日志

统一推荐字段：

- `requestId`
- `userId` / `adminId`
- `role`
- `path`
- `method`
- `status`
- `costMs`
- `tripId`
- `error`

安全要求：

- 不记录密码原文
- 不记录 token 原文
- 不记录对象存储 access key、secret key 等敏感配置原文
- 手机号、微信号等敏感字段写日志时需脱敏

## 10. 需求补缺默认规则

为保证开发可落地，以下内容作为默认规则写入技术方案：

- H5 若保留“删除行程”入口，后端按软删除/下线语义设计，不做物理删除
- H5 需补充“编辑行程页”，可复用发布页表单
- 历史行程联系方式按发布快照保存，不跟随用户资料变化
- 首页列表按发布时间倒序
- 我的发布按创建时间倒序
- 看板 v1 直接查 MySQL 聚合，不做缓存
