# 后端验证码邮件 HTML 设计

## 背景

`mn-backend` 当前在 [`mn-backend/internal/service/auth_service.go`](/Users/await/apros/moonick/mn-backend/internal/service/auth_service.go:387) 中直接拼接注册验证码邮件 HTML。现有模板信息完整，但视觉表现较为基础，缺少明确的信息层级、品牌感和验证码聚焦设计。

本次目标是在不改动后端发送逻辑的前提下，先产出 3 套可预览的 HTML 模板，供后续挑选和接入。

## 范围

本次产出放在 `mn-backend/htmls/`：

- `register-code-template-a1.html`
- `register-code-template-a2.html`
- `register-code-template-a3.html`

本次不包含：

- 后端 Go 模板接入
- 邮件发送逻辑重构
- 外链字体、图片、脚本资源

## 设计方向

采用用户确认的 `方案 A：清透品牌卡片`，保持统一基调：

- 蓝青渐变作为品牌识别
- 白色主卡片提升可读性
- 验证码作为视觉中心，必须一眼可见
- 中文邮件文案简洁直接，围绕 `邮箱验证码`、`5 分钟有效`、`非本人操作可忽略`

## 三套变体

### A1 主推版

强调平衡感和正式感：

- 顶部渐变品牌条
- 标题、说明、验证码卡片、提示区四段式布局
- 适合作为默认正式模板

### A2 极简版

强调效率和信息聚焦：

- 装饰减少
- 验证码模块尽量提前出现
- 适合以识别速度和兼容性为第一优先级

### A3 质感版

强调层次和完成度：

- 更细的边框、阴影、信息标签
- 保持邮件兼容前提下增加精致感
- 适合作为更有产品气质的候选版本

## 兼容策略

模板按邮件客户端优先设计：

- 使用 `table` 结构组织主布局
- 使用 `inline style`
- 不依赖现代 CSS 特性作为核心能力
- 不使用外链资源
- 保证在主流邮箱客户端中退化后仍可读

## 成功标准

满足以下条件视为本次设计产出完成：

- `mn-backend/htmls/` 下存在 3 套独立 HTML 文件
- 每套模板都能直接本地打开预览
- 每套模板都清晰呈现品牌名、验证码、有效期和安全提醒
- 三套模板属于同一品牌方向，但视觉层次有明确区分
