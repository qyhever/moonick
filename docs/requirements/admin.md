# 明叶同行 PC 管理端 PRD

---

## 产品概览

| 项目 | 内容 |
|------|------|
| 产品名 | 明叶同行管理后台 |
| 产品形态 | PC Web 管理端 |
| 使用对象 | 平台管理员 |
| 主要目标 | 支撑平台对行程和用户的基础运营管理 |
| 登录体系 | 独立管理员账号体系，与 H5 用户体系分离 |
| 权限模型 | v1 仅支持单管理员，不做角色细分 |
| 技术栈 | React + Vite + Ant Design |

---

## 范围说明

PC 管理端 v1 采用最小可用范围，优先满足日常运营场景，不扩展为复杂中台系统。

**纳入范围：**
- 管理员登录
- 首页轻量看板
- 行程管理
- 用户管理（只读查询）

**暂不纳入范围：**
- 多角色权限（RBAC）
- 操作日志
- 投诉/工单处理
- 公告配置
- 数据导出
- 趋势分析图表
- 用户封禁
- 用户资料编辑
- 营销活动配置

---

## 一、信息架构

PC 管理端采用后台常见布局结构：

- 顶部区域：系统名称、当前管理员信息、退出登录。
- 左侧菜单：固定菜单导航。
- 主内容区：页面内容展示与操作区域。

左侧菜单项：
- 首页
- 行程管理
- 用户管理

---

## 二、管理员登录页

### 2.1 表单项

| 字段 | 说明 |
|------|------|
| 管理员账号 | 必填，支持账号名登录 |
| 密码 | 必填，密码输入框，支持明文切换 |
| 登录按钮 | 校验通过后提交 |

### 2.2 登录规则

- 使用独立管理员账号体系。
- 与 H5 用户登录态完全隔离。
- 登录成功后进入 PC 管理端首页。
- 登录失败时展示错误提示。
- 登录状态可持久化，刷新页面后保持已登录状态。
- 未登录访问后台受保护页面时，统一跳转登录页。

---

## 三、首页（轻量看板）

> 登录成功后默认进入首页，用于展示核心运营数据和快捷入口。

### 3.1 展示内容

| 模块 | 内容 |
|------|------|
| 数据卡片 | 行程总数 |
| 数据卡片 | 今日新增行程数 |
| 数据卡片 | 用户总数 |
| 数据卡片 | 当前有效行程数 |
| 快捷入口 | 进入行程管理 |
| 快捷入口 | 进入用户管理 |

### 3.2 设计原则

- 只展示轻量运营信息，不做复杂 BI 图表。
- 优先保证信息清晰、加载快速、可直接跳转业务页。
- 风格遵循 Ant Design 后台管理规范，强调信息密度和可读性。

---

## 四、行程管理

> 行程管理是后台核心模块，用于查询、查看和编辑平台所有行程数据。

### 4.1 行程列表页

#### 筛选项

| 筛选项 | 组件类型 | 说明 |
|--------|----------|------|
| 行程类型 | 下拉选择 | 全部 / 车找人 / 人找车 |
| 起点 | 输入框 | 支持关键字搜索 |
| 终点 | 输入框 | 支持关键字搜索 |
| 出发日期 | 日期选择器 | 按出发日期筛选 |
| 行程状态 | 下拉选择 | `active / full / closed / expired` |
| 发布人 | 输入框 | 支持按昵称/手机号搜索 |
| 重置 | 按钮 | 一键清空所有筛选条件 |

#### 列表字段

| 字段 | 展示说明 |
|------|----------|
| 行程 ID | 唯一标识 |
| 行程类型 | 车找人 / 人找车 |
| 行程路线 | 起点 → 终点 |
| 出发时间 | 完整日期时间 |
| 座位/人数 | 展示当前人数信息 |
| 费用 | ¥XX / 面议 |
| 行程状态 | `active / full / closed / expired` |
| 发布人 | 昵称 + 手机号 |
| 发布时间 | 创建时间 |
| 更新时间 | 最后修改时间 |
| 操作 | 查看详情 / 编辑 |

#### 列表交互

- 支持分页查询。
- 分页参数沿用平台规范：`pageNum + pageSize`。
- 默认 `pageSize = 10`。
- 支持重置筛选条件。
- 无数据时展示后台空状态。

### 4.2 行程详情页

行程详情页需完整展示 H5 侧核心业务字段：

| 字段 | 展示说明 |
|------|----------|
| 行程 ID | 完整展示 |
| 行程类型 | 车找人 / 人找车 |
| 起点 | 完整展示 |
| 终点 | 完整展示 |
| 出发日期 | 完整展示 |
| 出发时间 | 完整展示 |
| 人数/座位 | 完整展示 |
| 人均费用 | ¥XX / 面议 |
| 微信号 | 完整展示 |
| 手机号 | 完整展示 |
| 备注 | 完整展示 |
| 行程状态 | `active / full / closed / expired` |
| 发布时间 | 完整展示 |
| 更新时间 | 完整展示 |
| 发布人信息 | 昵称、手机号、头像 |

### 4.3 行程编辑页

管理员可对行程进行全量编辑。

#### 可编辑字段

| 字段 | 说明 |
|------|------|
| 行程类型 | 可修改 |
| 起点 | 可修改 |
| 终点 | 可修改 |
| 出发日期 | 可修改 |
| 出发时间 | 可修改 |
| 人数/座位 | 可修改 |
| 人均费用 | 可修改 |
| 面议状态 | 可修改 |
| 微信号 | 可修改 |
| 手机号 | 可修改 |
| 备注 | 可修改 |
| 行程状态 | 可修改 |

#### 编辑规则

- 保存前统一执行表单校验。
- 校验规则可复用 H5 侧基础规则：
  - 起点和终点不能相同。
  - 出发时间不能早于当前时间。
  - 微信号与手机号至少填写一种。
- 点击保存时必须进行二次确认。
- 保存成功后提示操作成功，并返回详情页或列表页。
- 保存失败时展示具体错误信息。

#### 行程删除规则

- v1 默认不开放后台删除行程。
- 如需下线或终止行程，通过修改 `status` 实现。
- 若后续需要删除能力，应单独补充「软删除 / 硬删除」规则。

## 五、用户管理

> 用户管理 v1 定位为只读查询，不支持编辑和封禁。

### 5.1 用户列表页

#### 筛选项

| 筛选项 | 组件类型 | 说明 |
|--------|----------|------|
| 昵称 | 输入框 | 支持关键字搜索 |
| 手机号 | 输入框 | 支持精确或模糊查询 |
| 注册时间 | 日期范围选择器 | 按注册时间筛选 |
| 重置 | 按钮 | 一键清空所有筛选条件 |

#### 列表字段

| 字段 | 展示说明 |
|------|----------|
| 用户 ID | 唯一标识 |
| 头像 | 显示缩略图 |
| 昵称 | 对应字段 `nickname` |
| 手机号 | 完整展示 |
| 发布数量 | 该用户发布的行程数 |
| 收藏数量 | 该用户收藏的行程数 |
| 注册时间 | 完整展示 |
| 操作 | 查看详情 |

### 5.2 用户详情页

| 模块 | 展示说明 |
|------|----------|
| 基本资料 | 用户 ID、头像、昵称、手机号、注册时间 |
| 数据概览 | 发布数量、收藏数量 |
| 发布行程列表 | 展示该用户发布的行程摘要 |
| 收藏信息 | 可展示收藏数量或收藏摘要 |

### 5.3 用户管理规则

- v1 仅支持查看，不支持编辑。
- v1 不支持封禁、解封、删除用户。
- v1 不支持修改昵称、头像、手机号、联系方式。

## 六、前后端接口补充规范

> 当前需求重点是独立后台能力，建议使用后台专用接口，不直接复用 H5 展示接口。

### 6.1 后台认证接口

建议新增以下接口：

| 接口 | 说明 |
|------|------|
| `POST /admin/auth/login` | 管理员登录 |
| `POST /admin/auth/logout` | 管理员退出登录 |
| `GET /admin/auth/me` | 获取当前管理员信息 |

### 6.2 后台行程管理接口

建议新增以下接口：

| 接口 | 说明 |
|------|------|
| `GET /admin/trips` | 获取行程分页列表 |
| `GET /admin/trips/:id` | 获取行程详情 |
| `PUT /admin/trips/:id` | 更新行程信息 |

### 6.3 后台用户管理接口

建议新增以下接口：

| 接口 | 说明 |
|------|------|
| `GET /admin/users` | 获取用户分页列表 |
| `GET /admin/users/:id` | 获取用户详情 |
| `GET /admin/users/:id/trips` | 获取用户发布的行程列表（如详情页需要） |

### 6.4 接口设计原则

- 后台接口与 H5 接口鉴权隔离。
- 普通用户 token 不能访问后台接口。
- 后台列表接口统一支持分页。
- 后台编辑接口需要返回标准成功/失败结果。
- 行程状态枚举沿用现有定义：
  - `active`
  - `full`
  - `closed`
  - `expired`

## 七、前端实现建议

### 7.1 技术方案

| 项目 | 建议 |
|------|------|
| 构建工具 | Vite |
| 前端框架 | React |
| UI 组件库 | Ant Design |
| 路由 | React Router |

PC 管理端需具备以下基础能力：

- 路由鉴权
- 登录态持久化
- 后台基础布局（侧边栏 + 顶部栏 + 内容区）
- 通用分页表格
- 通用查询表单
- 通用详情展示区
- 通用编辑表单页
- 全局错误提示与空状态处理

## 八、验收标准

### 8.1 登录

- 管理员可通过独立账号密码登录后台。
- 未登录访问受保护页面时自动跳转登录页。
- 登录后刷新页面仍保留登录状态。

### 8.2 首页

- 登录成功后默认进入首页轻量看板。
- 首页可展示核心统计信息。
- 首页可快捷跳转到行程管理和用户管理。

### 8.3 行程管理

- 支持按类型、路线、日期、状态、发布人筛选行程。
- 支持分页查看行程列表。
- 支持查看任意行程详情。
- 支持编辑任意行程核心字段。
- 编辑前有二次确认。
- 编辑成功后数据正确更新。

### 8.4 用户管理

- 支持查询用户列表。
- 支持查看用户详情。
- 支持查看用户发布数量和收藏数量。
- 页面中不出现编辑、封禁、删除等用户处置操作。

---

## 九、与 H5 端的关系说明

- H5 端面向车主和乘客，是业务使用端。
- PC 管理端面向平台管理员，是运营管理端。
- H5 端和 PC 管理端共享同一业务数据，但登录体系和接口鉴权分离。
- PC 管理端优先服务于运营治理，不追求 H5 端的移动交互体验。
